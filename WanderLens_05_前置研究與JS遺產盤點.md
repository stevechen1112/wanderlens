# WanderLens 前置研究與 JS 遺產盤點

> 本文件為技術長視角的完整研究記錄，整合 WanderLens 五份企劃文件與 JoyShot JS 前身專案的技術分析，作為後續開發計畫的基礎依據。
>
> 核心原則：**JS 是可參考的細節來源，但 WanderLens 企劃才是主軸。不被 JS 牽著走，不反客為主。**

---

## 1. 專案全景定位

### 1.1 WanderLens vs JoyShot 的本質差異

| 維度 | JoyShot（JS 前身） | WanderLens（目標） |
|------|-------------------|-------------------|
| **商業本質** | 攝影師媒合服務 | 三層價值堆疊平台（服務→內容→數據） |
| **服務範疇** | 旅遊拍攝 + 活動紀錄為主 | 全場景（個人/家庭/婚禮/企業/空間/旅拍） |
| **供給結構** | 單一攝影師供給池 | 三供給池（攝影師 + 攝影棚 + 造型師） |
| **交付模式** | Google Drive 共用資料夾手動交付 | RAW 雲端留存 + AI 基本調色 + 48h SLA |
| **內容策略** | 無內容平台概念 | 相簿留存 + 公開授權 + SEO 內容池 + 場景數據 |
| **變現模式** | 單一服務抽成 | 服務抽成 + 精修加購 + 聯盟行銷 + 廣告 |
| **技術目標** | MVP 級別，能跑就好 | 可規模化、可跨境的平台架構 |
| **終極形態** | 區域性攝影預約網站 | 跨國旅拍網絡 + 內容資產平台 |

### 1.2 JS 可貢獻的價值

JS 雖然是陽春 MVP，但它已經**跑通了一條完整的攝影媒合服務鏈**，在以下方面提供了可參考的實作細節：

1. **三端分離架構**已驗證（客戶端 + 攝影師端 + 管理後台）
2. **訂單生命週期狀態機**已有基礎雛形
3. **金流整合**（綠界 ECPay）已實作完成
4. **攝影師行事曆管理**已有月曆 UI
5. **多語系架構**已有四語系框架
6. **外部服務整合**已有完整經驗（LINE/Google/SMS/Email）
7. **排程任務**已有 9 個 Cron Job 的業務邏輯

### 2.4 訂單狀態機對比

#### JS 現有狀態機（10 個狀態）

```
processing → pay_success → contact → uploaded → confirm-uploaded → transferred → close
                ↓
            pay_failed / cancel / auto_cancel
```

#### WanderLens 目標狀態機（17+ 個狀態）

```
Draft → PendingPayment → Paid → WaitingProviderContact → Confirmed → ReadyToShoot
  → ShootingStarted → ShootingEnded → UploadingRaw → AiProcessing → Delivered
  → RetouchRequested → Retouching → RetouchDelivered → Closed
  → Rematching / Rescheduled / Disputed / Refunded / Cancelled
```

#### 差異分析

| JS 狀態 | WanderLens 對應 | 差異說明 |
|---------|----------------|---------|
| processing | Draft → PendingPayment | JS 混合，WanderLens 拆分 |
| pay_success | Paid | ✅ 可參考 |
| contact | WaitingProviderContact → Confirmed | WanderLens 更細 |
| uploaded | UploadingRaw → AiProcessing | JS 無 AI 管線 |
| confirm-uploaded | Delivered | WanderLens 有交付 SLA |
| transferred → close | Closed | WanderLens 有清算帳本 |
| auto_cancel | Cancelled | ✅ 可參考排程邏輯 |
| ❌ | ShootingStarted/Ended | JS 無拍攝時間戳記 |
| ❌ | RetouchRequested/Retouching | JS 無精修流程 |
| ❌ | Rematching | JS 無重新媒合 |
| ❌ | Disputed/Refunded | JS 無爭議流程 |
| ❌ | Rescheduled | JS 無改期流程 |

### 2.5 業務流程對比

#### 預約流程

| 步驟 | JoyShot | WanderLens |
|------|---------|------------|
| 1 | 選服務類別 | 選拍攝類型 |
| 2 | 選日期/時間/時數/地點/人數 | 選拍攝配置（自動帶入建議） |
| 3 | - | 選時間與地點 |
| 4 | 搜尋攝影師列表 | 選服務人員（三池分別選） |
| 5 | 看攝影師詳情 | 確認總覽 |
| 6 | 結帳（填資料 + 付款） | 結帳 |

**關鍵差異**：WanderLens 的「選到即確定」vs JS 的「搜尋後預約」— JS 的攝影師時段管理已可參考，但 WanderLens 需要更嚴格的即時鎖定機制。

#### 交付流程

| 步驟 | JoyShot | WanderLens |
|------|---------|------------|
| 照片上傳 | 攝影師手動上傳到 Google Drive | 攝影師上傳 RAW 到平台雲端 |
| 交付方式 | Google Drive 連結給客戶 | AI 調色後上架至 App 相簿 |
| 交付時效 | 無 SLA | 48 小時 SLA |
| 照片留存 | Google Drive（攝影師擁有） | 平台雲端（平台擁有） |
| 相簿體驗 | 無 | App 相簿（留存主場） |

**關鍵差異**：這是 WanderLens 與 JS 最大的架構分歧點。JS 的 Google Drive 方式完全不符合 WanderLens 的「RAW 由平台雲端留存」前提。

### 2.8 已知問題與技術債（JS 需避免重蹈覆轍）

| 問題 | 影響 | WanderLens 對策 |

## 4. 技術決策建議

### 4.1 架構決策

| 決策項 | 建議 | 理由 |
|--------|------|------|
| 前端框架 | 評估 Vue 3 + Composition API | JS 用 Vue 2，但 WanderLens 是新專案，Vue 3 更長期 |
| 後端框架 | Spring Boot 3.x + Java 17+ | JS 的 Spring Boot 2.5.7 已過時 |
| 資料庫 | MySQL（沿用）+ Redis（新增） | JS 已驗證 MySQL，Redis 用於時段鎖定 |
| 物件儲存 | AWS S3 / Google Cloud Storage | 替換 JS 的 Google Drive |
| 認證 | Spring Security + JWT（獨立密鑰） | 替換 JS 的 MD5 + 密碼作密鑰 |
| API 文件 | SpringDoc OpenAPI 3 | 替換 JS 的 Swagger 2 |
| 金流 | 綠界 ECPay（沿用） | JS 已完整整合 |
| 通知 | LINE Notify + SMS + Email（沿用） | JS 已完整整合 |

### 4.2 RAW 上傳管線設計（WanderLens 核心挑戰）

JS 完全沒有此能力，需從零建設：

```
攝影師端 → 分段上傳 API → 物件儲存（S3/GCS）
                ↓
         JPEG 快路徑 → AI 調色 → 相簿上架（快速交付）
                ↓
         RAW 完整上傳 → 驗收 → 精修備用
```

關鍵設計要點：
1. **分段上傳 + 斷點續傳**：Multipart Upload
2. **JPEG 快路徑**：機身 JPEG 先上傳給 AI 調色，RAW 背景上傳
3. **批次驗收**：檔案數、容量、格式、可解碼性
4. **48h SLA 監控**：倒數告警 + 失敗重試
5. **上傳隊列**：網路中斷恢復

### 4.3 三池媒合引擎設計

JS 只有單池（攝影師），WanderLens 需三池交集：

```
攝影師池（行事曆 + 服務區 + 拍攝類型能力）
        ∩
攝影棚池（檔期 + 地點 + 棚型 + 可容納服務）
        ∩
造型師池（檔期 + 服務區 + 前置緩衝計算）
        ↓
即時可訂清單（選到即確定 → 時段鎖定）
```

關鍵設計要點：
1. **時段鎖定機制**：Redis 分散式鎖，防止併發預約衝突
2. **造型師前置緩衝**：自動計算（如 10:00 拍攝 → 造型師 08:30 到場）
3. **檔期交集計算**：三池行事曆即時交集
4. **選到即確定**：不需等待確認，直接鎖定

### 4.4 多層授權模型設計

JS 完全沒有此概念，需從零建設：

```
私密 → 連結分享 → 平台公開 → 攝影師作品集 → 平台行銷 → 商業合作
  ↑                                                        ↓
  └────────────── 可撤回（任一方撤回即下架）────────────────┘
```

關鍵設計要點：
1. **雙向授權**：消費者 + 攝影師都需同意
2. **未成年人特別處理**：預設更高隱私
3. **可撤回**：任一方撤回即立即下架
4. **SEO 索引聯動**：撤回時觸發下架流程

### 4.5 站內溝通管道設計（WanderLens 核心基礎設施）

#### JS 的缺陷

JS 時期讓消費者與攝影師透過 LINE / WhatsApp 等第三方通訊工具溝通，這存在嚴重的結構性缺陷：

| 問題 | 影響 |
|------|------|
| **平台無法得知溝通內容** | 爭議時無從舉證，各說各話 |
| **消費者無保障** | 攝影師可能私下要求加價、改條件，平台不知情 |
| **攝影師無保障** | 消費者可能私下要求超出服務範圍的內容 |
| **平台無保障** | 無法監控服務品質、無法發現違規行為 |
| **無法沉澱溝通數據** | 拍攝需求細節、客戶偏好等資料流失 |
| **無法跨境擴張** | 各國通訊工具不同（LINE 限亞洲、WhatsApp 限特定市場） |
| **隱私暴露** | 雙方手機號碼直接暴露給對方 |

#### WanderLens 站內溝通設計

WanderLens 必須建立完整的站內溝通管道，取代第三方通訊工具：

```
┌─────────────────────────────────────────────────────────┐
│                   站內溝通管道                            │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │ 站方 ↔ 攝影師│  │ 站方 ↔ 消費者│  │消費者 ↔ 攝影師│     │
│  │  (管理通道)   │  │  (客服通道)   │  │ (訂單通道)   │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
│         │                │                │             │
│         └────────────────┼────────────────┘             │
│                          ▼                              │
│              ┌──────────────────────┐                   │
│              │   溝通記錄留存中心     │                   │
│              │  （可追溯、可舉證）    │                   │
│              └──────────────────────┘                   │
└─────────────────────────────────────────────────────────┘
```

#### 三條溝通通道

| 通道 | 觸發時機 | 參與方 | 用途 | 限制 |
|------|---------|--------|------|------|
| **管理通道** | 平台營運需求 | 站方 ↔ 攝影師/造型師 | 審核通知、違規警告、政策公告、檔期提醒 | 攝影師端可見，消費者不可見 |
| **客服通道** | 消費者主動求助 | 站方 ↔ 消費者 | 訂單問題、退款爭議、品質申訴、使用協助 | 營運後台可見，攝影師不可見（除非需調查） |
| **訂單通道** | **訂單成立後** | 消費者 ↔ 攝影師/造型師 | 拍攝細節確認、地點討論、服裝建議、改期協商 | **僅在訂單付款成立後開啟** |

#### 訂單通道的關鍵設計

**核心原則：成交才能聯繫**

```
預約前（未付款）
  │
  │  ❌ 消費者與攝影師之間無任何直接溝通管道
  │  ❌ 攝影師聯絡資訊不暴露
  │  ❌ 消費者聯絡資訊不暴露
  │
  ▼
訂單成立（付款完成）
  │
  │  ✅ 系統自動開啟「訂單溝通室」
  │  ✅ 攝影師於 24 小時內透過站內溝通聯繫消費者
  │  ✅ 雙方透過站內溝通確認拍攝細節
  │  ✅ 所有溝通內容留存於平台
  │
  ▼
拍攝完成 → 結案
  │
  │  訂單溝通室轉為「唯讀」狀態
  │  仍可查看歷史記錄，但不可再發送新訊息
  │
  ▼
爭議發生時
  │
  │  營運可調閱完整溝通記錄作為裁決依據
```

#### 訂單通道與狀態機的整合

| 訂單狀態 | 溝通室狀態 | 可發訊息 | 說明 |
|---------|-----------|---------|------|
| Draft / PendingPayment | 未開啟 | ❌ | 預約前無溝通 |
| Paid | **自動開啟** | ✅ | 付款後立即開啟，啟動 24h 聯繫倒數 |
| WaitingProviderContact | 開啟 | ✅ | 攝影師應於此階段聯繫 |
| Confirmed → ReadyToShoot | 開啟 | ✅ | 雙方確認細節、改期協商 |
| ShootingStarted → ShootingEnded | 開啟 | ✅ | 拍攝中溝通（加時討論等） |
| UploadingRaw → Delivered | 開啟 | ✅ | 交付相關溝通 |
| Closed | **轉唯讀** | ❌ | 結案後僅可查看 |
| Disputed | **開啟 + 營運介入** | ✅ | 爭議處理，營運可調閱 |
| Cancelled / Refunded | **轉唯讀** | ❌ | 取消後僅可查看 |

#### 溝通功能需求

**基本功能**：
- 文字訊息
- 圖片傳送（拍攝參考圖、服裝照片、場地照片）
- 預約資訊卡片（自動帶入訂單資訊，避免重複描述）
- 系統訊息（自動記錄關鍵節點：訂單成立、攝影師已聯繫、拍攝開始等）
- 訊息已讀標記

**進階功能**（階段二+）：
- 範本訊息（攝影師常用回覆範本，如「請問拍攝當天有無特殊需求？」）
- 改期申請（透過溝通室發起，系統記錄並更新訂單）
- 加時討論（拍攝中透過溝通室發起，連動加時付款流程）
- 多語翻譯輔助（跨境階段，解決語言障礙）

**安全與治理**：
- 敏感資訊遮罩（手機號碼、Email、地址自動遮罩，防止私下聯繫）
- 違規關鍵字偵測（偵測「私下交易」「不加平台」等意圖脫鉤行為）
- 騷擾檢舉機制（任一方可檢舉不當訊息）
- 營運調閱權限（客服與爭議處理人員可調閱，需記錄調閱日誌）

#### 資料模型

| 物件 | 說明 |
|------|------|
| **Conversation** | 溝通室（類型：管理/客服/訂單，關聯訂單 ID，狀態：開啟/唯讀/關閉） |
| **Message** | 訊息（類型：文字/圖片/系統/範本，發送者，內容，已讀時間） |
| **MessageTemplate** | 範本訊息（角色、語系、內容） |
| **ConversationAccessLog** | 調閱日誌（調閱者、時間、原因） |
| **MessageReport** | 檢舉記錄（檢舉者、被檢舉訊息、原因、處理狀態） |

| 面向 | JoyShot | WanderLens |

| 爭議舉證 | ❌ 各說各話 | ✅ 可調閱完整記錄 |
| 聯絡資訊保護 | ❌ 直接暴露 | ✅ 站內遮罩 |

| 跨境適用 | ❌ 工具因地區不同 | ✅ 站內統一 |

## 7. 結論

### 7.1 JS 的真正價值

JoyShot 的價值不在技術先進（它確實陽春），而在於它**用實作驗證了攝影媒合服務的可行性**：
- 攝影師願意在低單價下接案（已驗證）
- 三端分離架構可行（已驗證）
- 綠界金流可運作（已驗證）
- 多語系框架可運作（已驗證）
- 排程通知邏輯可運作（已驗證）

### 7.2 WanderLens 的超越點

WanderLens 不是 JoyShot 的升級版，而是**完全不同的商業物種**：

1. **服務層**：三池媒合 vs 單池、RAW 雲端 vs Google Drive、AI 交付 vs 手動交付
2. **內容層**：相簿留存 + 公開授權 + SEO 內容池（JS 完全沒有）
3. **數據層**：場景標籤 + 行為事件 + 聯盟行銷（JS 完全沒有）
4. **跨境層**：雙向飛輪 + 多幣多語（JS 完全沒有）

1. **下一步**：基於本文件，制定階段一的詳細開發計畫
